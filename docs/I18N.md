# Internationalisation

Proposal Engine ships with **6 locales**:

| Code | Native name | Status                                                          |
| ---- | ----------- | --------------------------------------------------------------- |
| `en` | English     | source-of-truth                                                 |
| `es` | Spanish     | full coverage                                                   |
| `pt` | Português   | full coverage (refine with native review before public release) |
| `fr` | Français    | full coverage (refine with native review)                       |
| `de` | Deutsch     | full coverage (refine with native review)                       |
| `it` | Italiano    | full coverage (refine with native review)                       |

CI guards parity: [`src/i18n/coverage.test.js`](../src/i18n/coverage.test.js)
fails the build if any locale is missing a key that exists in `en`.

## Guardrails

- Never hardcode user-facing strings in components, tabs, editor controls, or export feedback.
- Every new label, toast, button text, or command entry must be added under `app`, `shared`, or the relevant namespace in `src/i18n/locales/en.js`.
- All other locales must include the same key before merging (enforced by `coverage.test.js`).

## Using `t()`

```js
import { t, withLang } from './i18n/translate.js';

t('en', 'tabs.client'); // → 'Client'
t('es', 'close.validityLabel', { date: '2026' }); // → 'Esta propuesta es válida hasta 2026'

const tt = withLang(lang); // curry for components
tt('app.exportPdf');
```

Lookup order: requested locale → `en` → `opts.fallback` → key (so
missing keys are visible in dev as their dotted path).

## Pluralisation

Driven by `Intl.PluralRules`:

```js
pluralize('en', 19, { one: '{n} country', other: '{n} countries' });
// → '19 countries'
```

## Locale-aware formatting

```js
import {
  formatCurrency,
  formatNumber,
  formatPercent,
  formatMonthYear,
  formatDateLong,
} from './i18n/format.js';

formatCurrency(12500, 'EUR', 'es'); // '12.500 €'
formatCurrency(1500.5, 'USD', 'en'); // '$1,500.5'
formatMonthYear(new Date(), 'en'); // 'April 2026'
formatPercent(0.95, 'en'); // '95%'
```

Supported currencies: `EUR`, `USD`, `GBP`, `BRL`, `MXN`, `CHF`.
Default: `EUR`. Override per-proposal via `state.currency`.

## Adding a new locale

1. Copy `src/i18n/locales/en.js` to `src/i18n/locales/<code>.js`.
2. Translate every value (keys must stay identical to `en.js`).
3. Update the `meta` block:
   ```js
   meta: { code: 'nl', label: 'Nederlands', nativeName: 'Nederlands', flag: '🇳🇱', direction: 'ltr' }
   ```
4. Register the import in `src/i18n/translate.js`:
   ```js
   import { nl } from './locales/nl.js';
   export const SUPPORTED_LOCALES = ['en', 'es', 'pt', 'fr', 'de', 'it', 'nl'];
   export const LOCALES = { en, es, pt, fr, de, it, nl };
   ```
5. Run `npm test` — `coverage.test.js` will fail until parity is
   reached.
6. Update this document.

## Browser auto-detection

`detectBrowserLang()` reads `navigator.languages`, normalises each
candidate (strips region, lower-cases), and returns the first
supported code. If none match, falls back to `en`.

`App.jsx` uses persisted `state.language` once available; only the
first-run experience defers to the browser.

| English      | Spanish                | Portuguese   | French                 | German          | Italian        |
| ------------ | ---------------------- | ------------ | ---------------------- | --------------- | -------------- |

| Lead         | Lead                   | Lead         | Lead                   | Lead            | Lead           |

untranslated unless local market practice requires otherwise.
